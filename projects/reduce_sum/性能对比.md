# Reduce Sum 性能对比

## 算子描述

- **计算公式**: `output = sum(x, axis=-1)`
- **输入**: x `[M, N]`
- **输出**: y `[M]`
- **数据类型**: float32
- **实现模式**: 两种 — Naive (每行 1 program) / Optimized (XBLOCK/XBLOCK_SUB/RBLOCK constexpr 循环)

## 瓶颈分析

当前 Triton 实现比 PyTorch 慢 3.1x，核心问题：

1. **xblock_sub=8 过小**: Optimized 模式 `xblock_sub = min(xblock, 8)`，每个子块仅处理 8 行。对于 1024 行的输入，需要 128 次循环迭代，每次只搬运 8×RBLOCK 的微小数据块，MTE2/MTE3 吞吐极低
2. **Optimized 不优于 Naive**: 因为 xblock_sub=8 几乎没有批处理收益，反而增加了 constexpr 循环的编译和调度开销
3. **未充分利用 UB**: 8 行 × RBLOCK 元素远未填满 96KB doublebuffer 槽位
4. **tl.sum(x, axis=1) 粒度太小**: 在 [8, RBLOCK] 的块上做 axis=1 sum，向量化效率低

## 优化方向

| 优化项 | 预期收益 | 说明 |
|-------|:-------:|------|
| 增大 xblock_sub 填满 UB | 高 | 按 `96KB / (RBLOCK * element_bytes)` 计算，float32+RBLOCK=1024 时 xblock_sub 可达 24，远大于 8 |
| 参考 vector_add UB 自适应策略 | 高 | 类似 `_optimal_block_size()`，根据 dtype 和 I/O 张量数自动计算 |
| RBLOCK 与 xblock_sub 联合调优 | 中 | 在 UB 容量内平衡 reduce 轴和 batch 轴的切分 |
| 减少循环迭代次数 | 中 | 更大的子块 → 更少的 constexpr 循环 → 更好的流水线利用率 |
