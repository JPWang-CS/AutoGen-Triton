## 9.1 NPU 编译选项

Triton-Ascend 提供 17 个 NPU 编译选项，通过 `triton.Config` 传入 autotune：

### 在 Config 中使用编译选项

```python
import triton

@triton.autotune(
    configs=[
        # 在 triton.Config 中传入编译选项
        triton.Config({"BLOCK_SIZE": 128, "multibuffer": True}),
        triton.Config({"BLOCK_SIZE": 256, "multibuffer": True}),
    ],
    key=["n_elements"],
)
@triton.jit
def my_kernel(...):
    pass
```

### 常用编译选项

| 选项 | 适用算子 | 说明 | 可选值 |
|------|---------|------|--------|
| `multibuffer` | 全部 | 开启流水并行（存算并行），**默认 true** | true/false |
| `unit_flag` | 全部 | Cube 搬出优化 | true/false |
| `tile_mix_vector_loop` | CV 融合 | Vector 循环切几份 | 如 [2,4,8] |
| `tile_mix_cube_loop` | CV 融合 | Cube 循环切几份 | 如 [2,4,8] |
| `enable_auto_bind_sub_block` | CV 融合 | 自动绑定子块 | true/false |
| `enable_hivm_auto_cv_balance` | CV 融合 | 自动 CV 负载均衡 | true/false |
| `sync_solver` | CV 融合 | 同步求解器 | true/false |
| `inject_barrier_all` | 全部 | 自动注入全操作屏障 | true/false |
| `inject_block_all` | 全部 | 自动注入全操作块 | true/false |
| `enable_linearize` | 全部 | 线性化 pass | true/false |
| `enable_nd2nz_on_vector` | CV 融合 | ND→NZ 布局变换 | true/false |

> **注**：VV 算子 = 只使用 Vector Core；CV 算子 = 同时使用 Cube Core + Vector Core（如含 `tl.dot` 的算子）。

### 常用环境变量

| 环境变量 | 作用 | 默认值 |
|---------|------|--------|
| `TRITON_ALL_BLOCKS_PARALLEL` | 自动将逻辑核数调整为物理核数，减少调度开销 | 未设置 |
| `TRITON_PRINT_AUTOTUNING` | 打印 autotune 最优配置信息 | 0 |
| `TRITON_CACHE_DIR` | Triton 编译缓存目录 | ~/.triton/cache |
| `TRITON_INTERPRET` | 解释器模式（慢但可调试） | 0 |
| `ASCEND_GLOBAL_LOG_LEVEL` | CANN 日志级别（0=DEBUG） | 3 |

## 9.2 昇腾扩展 API

### tl.dot_scaled

带缩放的矩阵乘法，用于注意力机制中的 Q@K^T * scale：

```python
# 标准: S = tl.dot(Q, K.trans()) * scale
# 扩展: S = tl.dot_scaled(Q, K, scale)  # 一步完成，可能更高效
```

### tl.inline_asm_elementwise

内联 Ascend Vector 汇编指令：

```python
# 用于实现 Triton 尚未内置的操作
result = tl.inline_asm_elementwise(asm_string, ...)
```

### tl.parallel / tl.sync_block

NPU 多核同步原语：

```python
@triton.jit
def my_kernel(...):
    # 并行区域
    tl.parallel(...)
    # 块内同步
    tl.sync_block()
    # 全核同步
    tl.sync_block_all()
```

### tl.gather / tl.scatter

高级索引操作：

```python
# 按索引收集
values = tl.gather(source, indices, axis=0)

# 按索引散射
tl.scatter(target, indices, values, axis=0)
```

## 9.3 调试方法

### device_print

在 kernel 内打印调试信息：

```python
@triton.jit
def debug_kernel(x_ptr, n, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    x = tl.load(x_ptr + offsets, mask=offsets < n)

    # 打印调试信息
    tl.device_print("pid:", pid, "offsets:", offsets, "x:", x)
```

**注意**：`device_print` 会严重影响性能，仅用于调试。

### device_assert

在 kernel 内进行断言检查：

```python
@triton.jit
def safe_kernel(x_ptr, n, BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n
    x = tl.load(x_ptr + offsets, mask=mask)

    # 断言：所有值应该为正
    tl.device_assert((x >= 0).all(), "Found negative values!")
```

## 9.4 性能分析

### 算子瓶颈分类

调优前先判断算子类型，不同类型适用不同优化手段：

| 算子类型 | 特征 | 瓶颈 | 有效优化手段 | 代表算子 |
|---------|------|------|------------|---------|
| **内存 bound** | 2~3 I/O, 极少计算 | GM 带宽 | 物理核绑定、大 BLOCK | add, copy |
| **计算 bound** | 多步计算、归约 | Vector 计算 | 标量退化避免、multibuffer | softmax, layernorm |
| **UB bound** | 大中间张量 | UB 容量 | XBLOCK/XBLOCK_SUB 二级切分 | reduce_sum |
| **CV 融合** | tl.dot + 后处理 | Cube/Vector 协调 | CV 负载均衡 | flash-attn |

### triton.testing.do_bench

测量 kernel 执行时间：

```python
import os
# 必须在 import torch_npu 之前设置
os.environ["ASCEND_GLOBAL_LOG_LEVEL"] = "3"  # 抑制 CANN 日志
os.environ["TRITON_BENCH_METHOD"] = "npu"    # NPU 精确计时

import triton.testing

# 测量执行时间（毫秒）
ms = triton.testing.do_bench(lambda: my_kernel[grid](x, y, n))

print(f"执行时间: {ms:.3f} ms")

# 计算 TFLOPS（注意: ms × 1e-3 转秒，不是 × 1e-6）
flops = 2 * M * N * K
tflops = flops / (ms * 1e-3) / 1e12
print(f"性能: {tflops:.2f} TFLOPS")
```

**关键**: NPU 上必须设 `TRITON_BENCH_METHOD="npu"` 才能获得准确计时。

### 静默采集: _silent_bench

`TRITON_BENCH_METHOD="npu"` 会触发 CANN profiler，每次 `do_bench` 打印 3-4 行日志。用 `_silent_bench` 包裹：

```python
import sys
from io import StringIO

def _silent_bench(fn, quantiles=None, warmup=10, rep=100):
    """静默执行 do_bench，抑制 CANN profiler 日志"""
    if quantiles is None:
        quantiles = [0.5, 0.2, 0.8]
    old_out, old_err = sys.stdout, sys.stderr
    sys.stdout = StringIO()
    sys.stderr = StringIO()
    try:
        result = triton.testing.do_bench(fn, quantiles=quantiles, warmup=warmup, rep=rep)
    finally:
        sys.stdout = old_out
        sys.stderr = old_err
    return result
```

**推荐模式**: 先采集所有数据存入变量，最后统一打印，避免日志混入结果。

### 带宽计算

```python
def calc_bandwidth_gbs(n_elements, element_bytes, ms):
    """do_bench 返回 ms，转秒用 ×1e-3"""
    return 3 * n_elements * element_bytes / (ms * 1e-3) * 1e-9

# 常见错误: ms * 1e-6 (结果偏高 1000 倍!)
```

### msprof Profiling

使用华为 msprof 工具进行详细的性能分析：

```bash
# 板上采集: 在 NPU 上运行并采集实际性能数据
msprof op python my_kernel.py

# 指令级仿真: 无需 NPU 设备，分析指令流水线
msprof op simulator --soc-version=Ascend910B3 python my_kernel.py

# 采集特定 kernel
msprof op --kernel-name=my_kernel --output=./prof_out python script.py
```

### 性能瓶颈模式

| 瓶颈 | 特征 | 原因 | 解决方案 |
|------|------|------|---------|
| **MTE2 Bound** | MTE2 占比 >> Vector | BLOCK_SIZE 太小 | 增大 BLOCK_SIZE, 开 multibuffer |
| **Vector Bound** | Vector 占比高 | 计算密集 | 优化计算逻辑 |
| **Scalar Fallback** | Scalar 占比异常高 | i32/i64 比较退化 | `.to(tl.float32)` 转浮点比较 |
| **Bubble** | Bubble 占比高 | 搬运计算串行 | 开 multibuffer 流水重叠 |

### 关键性能指标

| 指标 | 含义 | 优化方向 |
|------|------|---------|
| `aiv_vec_ratio` | Vector 计算时间占比 | 理想 >60%，增大 BLOCK_SIZE 提高计算访存比 |
| `aiv_scalar_ratio` | 标量计算时间占比 | 理想 <5%，减少整数比较退化为标量 |
| `aiv_mte2_time` | 数据搬运 GM→UB 时间 | 使用 multibuffer 优化 |
| `aiv_mte2_ratio` | 搬运占比 | 与 Vector 占比平衡 |
| `cube_util` | Cube 利用率 | 增大矩阵分块 |

### 优化决策树

```
性能不达预期
  ├── msprof 采集数据
  │   ├── MTE2 高? → 增大 BLOCK_SIZE / 确认 multibuffer=True
  │   ├── Scalar 高? → 查找整数比较，转 .to(tl.float32)
  │   ├── Bubble 高? → 开 multibuffer / 调整 Tiling
  │   └── Vector 高? → 检查计算逻辑是否冗余
  ├── UB overflow? → 减小 BLOCK_SIZE / 使用 XBLOCK_SUB
  └── coreDim 超限? → 增大 BLOCK_SIZE / TRITON_ALL_BLOCKS_PARALLEL=1
```

## 9.5 GPU 到 NPU 迁移常见问题

### 1. UB OVERFLOW

**症状**：编译时报错 `UB OVERFLOW`

**原因**：BLOCK_SIZE 过大，片上内存不够

**解决**：
```python
# 减小 BLOCK_SIZE
BLOCK_SIZE = 1024  # 尝试改为 512 或 256

# 或拆分为子块循环
for sub_start in range(0, BLOCK_SIZE, SUB_BLOCK):
    sub_data = tl.load(ptr + sub_start + tl.arange(0, SUB_BLOCK), ...)
    # 分段处理
```

### 2. coreDim 超限

**症状**：`coreDim=xxxx can't be greater than UINT16_MAX`，或核启动数远大于物理核数导致性能下降

**解决方法一：固定核数**
```python
grid = (aicore_num,)  # 而非 (triton.cdiv(M, BLOCK_M) * triton.cdiv(N, BLOCK_N),)
```

**解决方法二：TRITON_ALL_BLOCKS_PARALLEL**
```bash
# 编译器自动将逻辑核数调整为物理核数
export TRITON_ALL_BLOCKS_PARALLEL=1
```
> 仅当逻辑核间可并行时生效。编译器自动调整 grid 大小为物理核数，减少调度开销。

### 3. num_warps / num_stages 无效

**症状**：设置了 GPU 参数但无效果

**解决**：移除 `num_warps` 和 `num_stages`，它们是 GPU 概念

### 4. 标量退化（Scalar Fallback）

**症状**：Scalar 占比异常高，Vector 利用率低

**原因**：NPU Vector 单元不支持 i32/i64 比较运算，退化为标量执行

**触发场景**：
- `tl.where(int_index < limit, a, b)` — 整数条件
- `tl.where(cols < N, ...)` — cols 是 tl.arange (int32)

**不退化的场景**：
- `tl.load(ptr, mask=rmask, ...)` — mask 中的整数比较由 MTE2 引擎处理
- `tl.store(ptr, val, mask=rmask)` — 同上

```python
# ❌ int32/64 比较退化为标量
cols = tl.arange(0, N)  # int32
xbar = tl.where(cols < N, x - mean, 0.0)  # cols < N 退化！

# ✅ 转为 float32 避免退化
cols_f = cols.to(tl.float32)
xbar = tl.where(cols_f < N, x - mean, 0.0)  # float 比较 → vector
```

### 5. 数据类型不支持

**症状**：`tl.dot` 报错不支持 uint/float64

**解决**：检查 Triton-Ascend 数据类型支持表

| 操作 | 不支持的数据类型 | 影响 |
|------|----------------|------|
| `tl.dot` | uint、float64 | 编译错误 |
| Vector ADD | int64 | 退化为标量运算 |
| `tl.dot out_dtype` | int8, fp16 | 不支持 |

### 5. 尾轴未对齐

**症状**：性能远低于预期

**解决**：确保 BLOCK_SIZE 是分形大小（FP16/BF16 为 16）的倍数

```python
# ✅ 对齐
BLOCK_SIZE = 128  # 128 / 16 = 8

# ❌ 不对齐
BLOCK_SIZE = 100  # 100 / 16 = 6.25
```

## 9.6 性能优化技巧

### care_padding=False 减少同步

当 `tl.load` 的 mask 不能覆盖全部 BLOCK_SIZE 时，NPU 默认先用 Vector Core 填充未覆盖部分为 0（或其他值），再用 MTE2 指令搬运有效数据，两者产生同步依赖。如果未覆盖部分不影响后续计算，可跳过填充：

```python
# ❌ 默认行为：NPU 先用 Vector 填充 padding=0，再搬运数据，两者串行
data = tl.load(input + idx, mask=mask)

# ✅ 跳过 padding 填充，MTE2 搬运和后续计算可并行
data = tl.load(input + idx, mask=mask, care_padding=False)
```

> **适用条件**：未覆盖部分的值不影响后续计算结果（如后续会被 mask 过滤掉）。

### 借轴转置优化

当 tensor 尾轴大小不是对齐要求的倍数（如 shape=(2048, 3) 的 3）时，硬件自动补齐导致性能恶化。**借轴转置**技巧从长轴借一根对齐轴给短轴：

```python
# 原始: conv_state = [2048, 3], bfloat16
# 问题: 尾轴 3 不对齐（3 * 2 = 6 bytes < 32 bytes）

# 方案: 当成一维 load，然后通过 reshape+trans 借轴
conv_state = tl.load(conv_state_ptr + ...)  # 1D load, numel=2048*3=6144 对齐
# 长轴(2048)裂出一根对齐轴(16)借给短轴(3)
conv_state_T = conv_state.reshape(128, 16 * 3).trans().reshape(16, 3 * 128).trans().reshape(3 * 2048,)
```

> **适用条件**：`tensor.numel() * dtype_size` 是 256 bytes 的倍数。

### 先搬 UB 再 select（离散访存优化）

对于离散索引访问（如 gather），先连续搬运到 UB，再从 UB 中 select：

```python
@triton.jit
def pick_kernel(x_ptr, idx_ptr, y_ptr, M: tl.constexpr, N: tl.constexpr):
    pid = tl.program_id(0)
    rn = tl.arange(0, N)
    idx = tl.load(idx_ptr + rn)
    mask = idx < M

    # ❌ 直接按索引 load（离散访存，多次小批量搬运，慢）
    # val = tl.load(x_ptr + idx, mask=mask)

    # ✅ 先全部搬到 UB，再 gather
    rm = tl.arange(0, M)
    x_shared = tl.load(x_ptr + rm)  # 一次性连续搬运到 UB
    val = tl.gather(x_shared, idx, 0)  # 在 UB 中 select

    tl.store(y_ptr + rn, val, mask=mask)
```

### VV 与 CV 算子的对齐差异

| 算子类型 | 说明 | 尾轴对齐要求 |
|---------|------|------------|
| VV (Vector-Vector) | 只使用 Vector Core | 32 字节 |
| CV (Cube-Vector) | 使用 Cube Core + Vector Core | 512 字节 |

```python
# VV 算子: float16 尾轴 >= 16 (16 * 2 = 32 bytes)
# CV 算子: float16 尾轴 >= 256 (256 * 2 = 512 bytes)
```

### 存算并行（Multibuffer）

NPU 支持"数据搬入/计算/搬出"并行。默认开启 `multibuffer=True`，通过 for 循环增加 Tiling 可让多缓冲生效：

```python
@triton.jit
def optimized_kernel(x_ptr, out_ptr, N, BLOCK_SIZE: tl.constexpr, BLOCK_SUB: tl.constexpr):
    pid = tl.program_id(0)
    base = pid * BLOCK_SIZE
    num_sub = tl.cdiv(BLOCK_SIZE, BLOCK_SUB)

    for i in range(num_sub):
        offsets = base + i * BLOCK_SUB + tl.arange(0, BLOCK_SUB)
        mask = offsets < N
        x = tl.load(x_ptr + offsets, mask=mask)
        result = x * 2.0 + 1.0  # 计算与下一次搬运并行
        tl.store(out_ptr + offsets, result, mask=mask)
```
